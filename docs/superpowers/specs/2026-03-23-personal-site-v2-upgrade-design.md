# 个人网站第二版升级设计方案

日期：2026-03-23

## 升级目标

本次升级在第一版交互式静态个人站的基础上，解决以下五个核心问题：

1. `Hero -> Projects` 以及 `Projects -> 下半段` 过渡不够丝滑
2. `Notes` 仍然像占位内容，不像真正的文章系统
3. `Photos` 过于朴素，缺乏第二视觉高潮
4. 首屏动效密度和抓人能力不足
5. 站点整体长度与探索深度不够

升级后的站点应从“一个完成度不错的交互首页”，提升为“一个具有主舞台、内容系统、视觉第二层的完整数字空间”。

## 本次升级后的产品定位

### 保持不变的核心定位

- 交互体验优先于普通简历叙述
- 首页仍然是全站主舞台
- `Claw` 继续作为第三人称叙事视角
- 视觉主方向仍为 `A2 / 暗铬实验室`

### 明确增强的方向

- 首页前半段动效密度明显提高
- `Projects` 章节从单段炫技升级为完整章节
- `Notes` 从摘要块升级为内容系统
- `Photos` 从平面占位升级为“短暂 3D 展示 + 长期档案承载”
- 网站整体长度不靠注水，而靠层次和内容网络自然增长

## 总体架构升级

第二版采用：

- 首页作为主舞台
- `Notes` 作为唯一新增的独立静态文章页体系

站点结构从：

- `/` 首页（单页主舞台）

扩展为：

- `/` 首页
- `/notes/<slug>/` 每篇随想独立页

本次升级不做：

- `Projects` 独立详情页
- `Photos` 独立图库页
- 归档页 / 标签页 / 年份页
- CMS / 后台

## 首页第二版结构

首页从第一版的：

- Hero
- Signals
- Projects
- Notes
- Photos
- Contact

升级为以下节奏结构：

1. 首屏启动序章
2. 首屏交互回应
3. 连续解构层
4. Projects Prelude 预热段
5. Projects 主走廊
6. Projects Outro 退场段
7. Notes 时间线
8. Photos 3D 环形开场
9. Photos 档案瀑布流
10. Contact 尾声

本次升级的重点不只是增加模块，而是强化模块之间的连接关系与节奏衔接。

## 首屏升级设计

### 目标

解决“第一页交互式动效不够多、抓眼不够强”的问题。

### 动效策略

首页前 10-15 秒采用：

- 压迫式抓人
- 随后迅速收住

不采用持续高压整页轰炸，而采用：

- 高密度多层启动
- 短时间叙事展开
- 再顺滑进入首页主线

### 首屏分为四个相位

1. `0-3 秒：启动压迫`
   - 背景空间、主物件、辅助信号、文案同时启动
   - 形成“系统正在接管屏幕”的事件感

2. `3-6 秒：主物件回应`
   - 鼠标移动、首次滚动或短暂停留触发主物件结构反馈
   - 包括层片错位、边缘光激活、局部结构滑移

3. `6-10 秒：叙事展开`
   - Claw 的播报式文案分段出现
   - 主物件显露更深层结构

4. `10-15 秒：解构进入下一层`
   - 主物件拆解出 `Projects / Notes / Photos` 三个信号方向
   - 顺滑进入首页主线

### 首屏设计原则

- 动效密度上调
- 同时允许更多层同时运作
- 但必须保持流畅，不接受明显掉帧和节奏混乱
- 首屏高潮必须在 10-15 秒内收住

## Projects 章节重构

### 目标

解决：

- 滑到 `Projects` 时不够丝滑
- 从 `Projects` 继续往下滑时也不够丝滑

### 新结构

`Projects` 章节升级为完整三段式结构：

1. `Prelude` 预热段
2. `Main Corridor` 主走廊
3. `Outro` 退场段

### Prelude 预热段

职责：

- 将用户从首页装置逻辑引导到项目展示逻辑
- 三个信号卡开始收拢
- 背景视觉开始聚焦
- 项目编号和第一张项目轮廓先被点亮

用户感受应为：

- 正在被带向一个核心展区

### Main Corridor 主走廊

职责：

- 保留整站唯一一次重型 `pin + scrub` 章节
- 三个项目在空间中前后切换焦点
- 当前项目负责接管舞台，其余项目退后

相较第一版新增要求：

- 主项目切换时要有明确“接管”感
- 不只是简单放大/缩小，而是焦点、层级、视觉权重一起切换

### Outro 退场段

职责：

- 将用户从重型项目章节平滑送回阅读模式
- 景深压平
- 透视消退
- 背景光效减弱
- 下一章节的阅读逻辑提前显影

这是第二版实现丝滑感的关键。

## Notes 体系升级

### 首页中的 Notes

首页中的 `Notes` 不再承载全文，而只承担：

- 时间定位
- 标题展示
- 摘要展示
- 进入全文动作

结构为：

- 时间线
- 节点摘要
- 进入独立文章页

### Notes 的视觉方向

不是博客列表，而是：

- 档案时间线
- 被整理过的思考轨迹
- 被记录下来的时刻

### 首页中的 Notes 动效

只做轻量阅读型动效：

- 细线显影
- 节点轻 reveal
- 焦点节点轻微提亮

不做：

- pin
- scrub
- 重型 3D

### 独立文章页结构

每篇 `Notes` 独立页应包含：

1. 页首
   - 标题
   - 日期
   - Claw 的一句开场旁白
   - 返回首页入口

2. 正文区
   - 真正可读的文章排版

3. 阅读辅助
   - 轻量进度线或时间标尺

4. 页尾导航
   - 上一篇 / 下一篇
   - 返回首页时间线

### 独立文章页动效原则

- 保持克制
- 重点是阅读，不是表演
- 仅保留页首进入、段落 reveal、轻量进度反馈

## Photos 体系升级

### 总体结构

`Photos` 第二版明确拆成两层：

1. 短暂 3D 环形开场
2. 档案瀑布流

### 3D 环形开场

目标：

- 作为第二个视觉高潮
- 抓住用户眼球
- 但不承担长期内容承载

设计原则：

- 像展台，不像转盘
- 环形结构短暂出现
- 某些照片前推成为焦点
- 随后快速让位

三阶段：

1. 环形浮现
2. 焦点前推
3. 解散归档

明确禁止：

- 无限旋转
- 大量照片持续乱飞
- 整段持续高强度炫技

### 档案瀑布流

目标：

- 作为长期扩展的照片承载结构
- 让网站变长和变丰富

视觉方向：

- 像观察样本墙
- 不是普通图库 feed

结构特征：

- 尺寸有主次
- 排布有呼吸感
- 局部可错位
- 不做密不透风的瀑布流

动效策略：

- 首次出现时轻 reveal
- hover 时轻提亮、轻微放大、信息条显现

## 站点丰富度增长方式

第二版网站变长不靠 filler，而靠四条真实增长线：

1. 首屏序章更完整
2. Projects 章节前后缓冲更完整
3. Notes 时间线 + 文章页形成内容系统
4. Photos 从装置升级为可持续扩展的档案层

### 目标节奏比例

- 高强度段：2 个
  - 首屏前半段
  - Projects 主走廊

- 中强度段：2 个
  - Projects 预热 / 退场
  - Photos 3D 开场

- 低强度段：其余内容区
  - Notes 时间线
  - Photos 档案瀑布流
  - Contact

这样网站长度增加，但高压段不会失控。

## 技术架构调整

### 保持不变

- `Vite + 原生 HTML/CSS/JS + GSAP`

### 新增层次

1. `Notes` 独立文章页静态页面结构
2. 升级后的内容模型
3. 更细粒度的首页动效模块
4. 用于生成文章页的轻量静态页面生成策略

### 内容模型升级方向

`notes` 至少需要支持：

- `slug`
- `title`
- `date`
- `summary`
- `clawIntro`
- `body`
- `previous`
- `next`

### 首页动效拆分方向

第二版首页建议拆成：

- `hero-sequence`
- `hero-signals-transition`
- `projects-prelude`
- `projects-corridor`
- `projects-outro`
- `notes-timeline`
- `photos-ring`
- `photos-waterfall-reveal`

不再将首页后半段所有内容统一塞进单个 reveal 模块。

## 性能与工程边界

尽管本次允许明显提升动效密度，但不接受：

- 明显掉帧
- 大面积布局抖动
- 用户滚动断裂
- 手机端发热严重的重型实现

继续坚持：

- 只用 `transform / opacity` 作为主要动画手段
- `Projects` 仍然是整站唯一真正的重型滚动主段
- `Photos` 的 3D 环形必须短
- `Notes` 时间线不做重型滚动
- 图片资源按用途和尺寸分级

## 本轮实现边界

### 本轮必须做

1. 首屏重构
2. `Hero -> Projects` 节奏重构
3. `Notes` 时间线 + 独立文章页
4. `Photos` 3D 环形开场 + 档案瀑布流
5. 首页长度与探索层次提升

### 本轮明确不做

1. `Projects` 独立详情页
2. `Photos` 独立图库页
3. 文章归档页 / 标签页 / 年份页
4. CMS / 后台
5. 大规模内容平台化

## 推荐实现顺序

建议第二版按以下顺序实施：

1. 重构内容数据结构
2. 先打通 `Notes` 独立文章页
3. 重构首页前半段与 Projects 章节
4. 最后做 `Photos` 3D 环形开场 + 瀑布流

这样即使中途停下，也会优先获得最有价值的结构提升。

## 验收标准

第二版升级完成后，应满足：

1. 首屏前 10-15 秒明显比第一版更抓人
2. `Hero -> Projects` 和 `Projects -> Notes` 过渡显著更丝滑
3. 首页长度和探索层次明显增加
4. `Notes` 可进入独立文章页
5. `Photos` 既有短暂高光，又能长期承载内容
6. 动效密度提高，但仍保持流畅
7. reduced-motion 与移动端仍有合理退化

## 下一步

本设计确认后，下一阶段应：

1. 写新的实现计划
2. 明确首页与文章页的文件拆分
3. 明确内容模型与生成方式
4. 再进入实现阶段
