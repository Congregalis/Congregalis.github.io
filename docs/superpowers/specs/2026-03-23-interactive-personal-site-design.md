# 交互式静态个人网站设计方案

日期：2026-03-23

## 目标概述

本项目将构建一个可部署在 GitHub Pages 的纯静态个人网站，重点使用 GSAP 构建具有记忆点的交互式动效体验。

站点目标优先级如下：

1. 交互体验型：先让访客愿意继续探索
2. 作品展示型：随后让访客看到项目能力和实现质量
3. 个人品牌型：最后再建立对站主本人的认识

主要观众优先级如下：

1. 泛互联网路人 / 朋友
2. 同行开发者 / 设计师
3. 潜在客户 / 合作方
4. 招聘方 / 面试官

首页在访客进入后的 30-60 秒内，最优先触发的行为如下：

1. 继续往下探索
2. 记住站主本人及其气质
3. 在合适时机产生联系意愿

现有内容库存：

- 约 3 个可展示项目
- 后续会持续补充随想内容
- 后续会逐步加入照片内容

## 体验方向

### 视觉方向

主视觉方向采用 `A2 / 暗铬实验室`。

该方向是以下两种气质的混合：

- 主体采用“展陈装置风”的空间感、材质感和层叠关系
- 骨架借用“信号实验室风”的系统感、编号感和技术气质

混合比例采用：

- 以展陈装置感为主体
- 以系统感为结构和暗纹

设计原则如下：

- 不做大面积赛博 HUD 或高频荧光界面
- 系统感仅作为骨架和秩序来源
- 高级感来自材质、节奏、比例、光感，而不是特效数量

### 结构方向

站点结构采用 `A + B` 混合方案：

- `A / 悬浮核心体 + 分层展开` 作为全站主骨架
- `B / 纵深走廊 + 章节推进` 仅在项目区插入一次重型章节

该结构的核心思想是：

- 用稳定、清晰的首页装置建立吸引力
- 在项目区集中释放一次最强记忆点
- 在其余区域快速回落到可读、可浏览的节奏

纵深走廊明确放置于：

- `Projects / 项目区`

不采用的策略：

- 不把全站都做成高强度 pin + scrub 体验
- 不让每个模块都使用同等级重交互
- 不将网站做成单纯炫技的 demo 集合

## 信息架构

整站为单页沉浸式体验，核心区块如下：

1. Hero / 悬浮核心体
2. Signal Layer / 入口解构段
3. Projects Corridor / 项目纵深走廊
4. Notes / 随想区
5. Photos / 照片区
6. About / Contact / 尾声与联系区

### 1. Hero / 悬浮核心体

内容极简，仅包含：

- 站主姓名
- 一句简短定位
- 一条轻量引导语

视觉上展示一个暗铬、烟熏玻璃、弱内发光的悬浮主物件。

职责如下：

- 第一时间建立站点气质
- 把访客从“浏览网页”切换到“进入一个装置”
- 提供足够低的信息密度，鼓励继续探索

### 2. Signal Layer / 入口解构段

当用户首次向下滚动时，Hero 不直接退出，而是逐步解构为三个内容信号层：

- Projects
- Notes
- Photos

职责如下：

- 把首页从“装置入口”转换成“内容世界”
- 提示后续内容结构
- 维持探索感，而不是使用传统导航式突兀切换

### 3. Projects Corridor / 项目纵深走廊

该章节是全站唯一重型滚动章节。

展示方式：

- 使用 pin + scrub 构建短时纵深走廊
- 3 个项目在空间前后层级中切换焦点
- 当前项目居中、最大、最清晰
- 非当前项目以透视缩放、亮度压低、轻微模糊退到前后景

每个项目在走廊内仅展示最关键的信息：

- 项目名称
- 一句价值描述
- 一个进入详情的动作入口

职责如下：

- 让访客在较短时间内对 3 个项目建立记忆
- 将全站最强动效集中在最有证明力的内容上
- 在“酷炫”与“有东西”之间建立可信联系

### 4. Notes / 随想区

随想区采用较安静的纵向条目流结构。

每条内容建议包含：

- 标题
- 日期
- 2-3 行摘要

职责如下：

- 给访客提供第二层理解站主的方式
- 承接项目区之后的节奏回落
- 作为未来持续更新的内容层

不采用的策略：

- 不做重型滚动叙事
- 不做章节级 3D 交互
- 不让随想区与项目区抢戏

### 5. Photos / 照片区

照片区采用有层次的编排带，而非传统九宫格。

结构特征如下：

- 纵向错位或轻量带状排布
- 像空间中被挂起的印样或样张
- 图片本身是重点，动效只做辅助

职责如下：

- 提供作品与文字之外的感知线索
- 让站主的审美偏好通过影像自然外显
- 为后续内容扩展预留空间

### 6. About / Contact / 尾声区

尾声区以低调、精致、平静的方式收口。

建议包含：

- 一小段关于站主的文字
- 当前关注的主题
- 联系方式与外链

职责如下：

- 给已经完成探索的访客提供进一步连接方式
- 避免网站最后塌成普通表单页或简历尾页

## 动效设计

整体动效强度选择为 `中强度`。

动效策略应遵守如下原则：

1. 80% 时间保持可读
2. 20% 时间制造记忆点
3. 全站只有一个真正重型的滚动章节
4. 其余区域以质感、节奏和层次为主，不做持续炫技

### Hero 动效

采用一条主 `gsap.timeline()` 管理首屏入场。

入场阶段建议包含：

- 主物件由压缩态展开
- 背景层缓慢显影
- 文本晚半拍出现

鼠标交互仅保留轻微跟随：

- 使用 `gsap.quickTo()` 控制 `x/y`
- 幅度较小
- 目的为增强“悬浮感”，而非制造强交互噱头

### Signal Layer 动效

使用短时间线与 `ScrollTrigger` 结合。

阶段表现为：

- 主物件分裂
- 图层错位
- 后退、解构
- 三个内容信号层浮出

仅使用低风险动画属性：

- `transform`
- `opacity` / `autoAlpha`
- 必要时少量 `filter`

避免使用布局重排型属性。

### Projects Corridor 动效

项目区是全站唯一重型章节。

实现建议如下：

- 使用单独的 `ScrollTrigger`
- 开启 `pin + scrub`
- 持续约 1.5-2.5 屏滚动长度
- 使用一个主时间线推进三个项目的焦点切换

明确限制：

- 全站仅此一处使用此强度
- 不将同类重型交互复制到随想区或照片区
- 移动端对该章节进行明显降级

### Notes / Photos / Footer 动效

使用轻量 reveal 动画即可。

建议使用：

- `autoAlpha`
- `y`
- `stagger`

hover 效果也应轻量：

- 微缩放
- 亮度变化
- 信息条显隐

不得使用的模式：

- 高频 3D 透视
- 大范围旋转
- 全区域持续跟随鼠标
- 多个重型章节并行存在

## 视觉系统

### 视觉关键词

- 暗铬
- 烟熏玻璃
- 冷石材
- 实验室标尺
- 展陈灯光

核心精神：

- 冷
- 准
- 克制

### 颜色系统

颜色比例建议如下：

- 80% 深色基底
- 15% 金属 / 雾面中间层
- 5% 冷色信号强调

色彩方向如下：

- 背景使用偏蓝灰的深色，而非纯黑
- 表面使用深石墨与暗金属灰
- 亮面材质使用雾银与烟白
- 强调色仅使用单一偏冷青绿色

强调色用途限制如下：

- 焦点提示
- 边缘光
- 编号与标尺
- hover 响应

不得大面积铺设。

### 字体系统

建议采用三套字体分工：

- 展示标题：带雕塑感的 serif，例如 `Instrument Serif`
- 正文：干净克制的 sans，例如 `Manrope`
- 编号 / 标签 / 系统标记：`IBM Plex Mono`

目标如下：

- 标题负责气质
- 正文负责阅读
- monospace 负责系统秩序

### 版式系统

桌面端采用大留白、非对称编排，以及持续出现但不过度抢眼的编号和标尺系统。

版式特征如下：

- 大标题不必总居中
- 卡片和图片存在控制良好的错位
- 分割线极细
- 边框存在感弱
- 圆角存在但偏硬，不做软萌玻璃 UI

### 材质与光感

材质使用原则：

- 暗铬反射
- 烟熏玻璃
- 半透明叠层
- 弱内发光

这些材质仅应用于关键层级，不做全站泛化。

## 文案系统

### 叙事视角

整站文案采用：

- 第三人称叙事
- 由助手 `Claw` 作为站内讲解员进行介绍

Claw 的角色不是可爱型吉祥物，而是：

- 冷面
- 专业
- 略带攻击性幽默
- 具有策展助手或系统代理人气质

Claw 负责：

- 介绍站主
- 引导用户穿过不同章节
- 在合适位置加入轻度吐槽和幽默

### 文案语气原则

建议关键词如下：

- 第三人称
- 冷面
- 精准
- 略带吐槽
- 不卖萌
- 不自嗨
- 不油腻

控制原则如下：

- 首屏和章节过渡处，Claw 存在感最强
- 项目区保持第三人称，但吐槽克制，确保信息清楚
- 随想和照片区更像档案或观察记录旁白
- About / Contact 区用低调但带人格的方式收口

### 示例方向

以下仅用于定调，不是最终定稿：

- “我是 Claw，XXX 的助手。”
- “我负责替他整理作品、记录念头，以及收拾那些差点被他做得太过火的东西。”
- “下面这些是他目前拿得出手的项目。数量不算多，好在还没烂到需要靠排版撑场面。”
- “如果你已经看到这里，Claw 判断你大概率不是随便路过。”

## 技术架构

### 技术选型

建议使用：

- `Vite`
- 原生 `HTML / CSS / JavaScript`
- `GSAP`

不建议在第一版使用：

- React
- Next.js
- Astro

原因如下：

- 本项目核心在于视觉与动效编排，而非复杂状态管理
- 原生 DOM + GSAP 对时间线和结构控制更直接
- GitHub Pages 部署更简单稳定

### 前端模块划分

建议按逻辑拆分脚本：

1. `main`
2. `hero`
3. `signals`
4. `projects-corridor`
5. `reveals`
6. `motion-config`

模块职责如下：

- `main`：初始化、媒体查询、动画注册
- `hero`：首屏时间线和鼠标轻跟随
- `signals`：入口解构段
- `projects-corridor`：项目纵深走廊
- `reveals`：轻量 reveal 动效
- `motion-config`：持续时间、easing、断点、reduced motion 配置

## 响应式与降级策略

### 桌面端

保留完整体验：

- 悬浮核心体
- 分层解构
- 项目纵深走廊
- 轻量 hover
- 细微鼠标跟随

### 平板 / 小桌面

保留整体结构，但降低：

- 景深幅度
- 位移幅度
- 视觉复杂度

### 移动端

保留同一气质，但重写互动强度：

- 弱化或取消完整纵深透视
- 项目区改为轻量 pin 或卡片分段切换
- 去除鼠标跟随
- 保证浏览流畅优先于完整还原桌面动效

原则如下：

- 移动端不是桌面端的缩小版
- 而是同一气质的另一种演法

## 性能边界

### 动画性能原则

- 优先动画 `transform`
- 优先动画 `opacity` / `autoAlpha`
- 避免使用 `top / left / width / height`

### 性能控制措施

- `will-change` 仅赋予关键元素
- 鼠标交互使用 `gsap.quickTo()`
- 仅在项目区使用一次重型 `pin + scrub`
- Notes / Photos 使用批量 reveal，而非大量复杂触发器
- 图片资源必须压缩

### 可访问性

必须支持：

- `prefers-reduced-motion`
- 键盘可达
- 动画失败时内容仍然成立
- 暗色界面下的足够对比度

在 reduced motion 下：

- 去掉鼠标漂移
- 去掉项目走廊重景深
- 退化为静态分段展示或极短动画

## 部署方案

部署目标为 GitHub Pages。

推荐方式：

1. 使用 `Vite` 构建
2. 配置正确的 `base` 路径
3. 输出产物到 `dist`
4. 使用 GitHub Actions 自动部署到 `gh-pages`

## 验收标准

设计与实现完成后，应至少满足以下标准：

1. 首屏打开 3 秒内建立明确气质
2. 用户可在 30-60 秒内自然进入项目区
3. 桌面端项目走廊流畅
4. 移动端不保留危险的高负载表现
5. 去掉动画后内容结构仍然成立
6. 用户即使不进行任何额外交互，也能顺畅完成浏览

## 非目标

本次第一版不追求以下内容：

- 复杂 CMS
- 服务端渲染
- 全站多页面复杂跳转
- 过量 3D 场景
- 高频粒子特效
- 为炫技而炫技的章节堆叠

## 后续建议

在本设计确认后，下一阶段应编写实现计划，明确：

- 页面与模块拆分
- 动效时间线拆分
- 内容占位与真实内容注入顺序
- 响应式实现细节
- 部署与验证流程
