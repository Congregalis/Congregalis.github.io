最近在做一个备面模块，流程其实不复杂：先做 retrieval，再把上下文喂给模型，让它输出一组结构化面试题。结果跑出来一看，上面那块“生成结果”里，模型写得头头是道；下面真正落盘到题目列表里的，却变成了两道朴素得像没睡醒的 fallback 题。

这事第一眼看着像模型抽风，仔细一扒，根本不是。模型内容其实没错，错的是它没按系统要求的格式说话。

## 问题往往不在“答得对不对”，而在“有没有按接口交付”

这次项目里，后端真正想要的是一个裸 JSON：

```json
{"questions":[...]}
```

结果模型吐出来的是这个味儿：

```json
{
  "questions": [
    {
      "id": 1,
      "type": "technical",
      "content": "针对腾讯元宝在线产品的 Agent Runtime 场景，请你设计一套可落地的 Memory 体系……"
    }
  ]
}
```

看起来几乎一样，对吧？但前面那一层 ```` ```json ```` 围栏一包，解析器直接就炸了。系统做的是严格 JSON 反序列化，不是阅读理解。它不会夸你“语义上差不多”，它只会冷冰冰地告诉你：这玩意不是合法 JSON。

然后事情就很滑稽了。流式面板上展示的是模型原始输出，所以你看到的是一套挺像样的问题；真正保存到 `session.questions` 里的，因为解析失败，只能走 fallback，最后变成“请说明你在 rag 相关项目中的关键决策和权衡”这种模板题。

这就是很多 AI 应用最容易忽略的一点：模型不是同事，不会自己体会“你应该懂我意思”。你给它的是生成任务，系统收的是接口结果。中间只要有一点格式漂移，前面再聪明，后面也得掉坑里。

## 提示词不是合同，最多算口头劝导

很多人遇到这种问题，第一反应就是继续补 prompt：

- 不要输出 markdown
- 不要加代码块
- 只返回 JSON
- 严格遵守格式

这当然有用，但也就那样。提示词本质上是在“提高概率”，不是在“建立约束”。你可以让模型更倾向于配合，但你不能指望它像一个受类型系统约束的函数一样稳定。

这事就像你跟一个很聪明但很随性的同事说：“麻烦你把数据按 CSV 给我。”他说“没问题”，然后转头给你发了个排版精美的表格截图。内容也许是对的，但系统根本吃不进去。

所以只靠 prompt，迟早还会再摔一次。不是今天多一个围栏，明天就可能多一句“以下是结果”。模型越会说，越容易在这种地方给你加戏。

## 真正靠谱的做法，是把“格式”从建议变成约束

这类问题一般有三层解法，强度从弱到强。

第一层是解析器容错。比如在进入 JSON 解析前，先把常见的 ```` ```json ```` 围栏剥掉，再提取首个合法 JSON 块。这个办法很实用，成本也低，适合先止血。像这次项目，光做这一层，模型那两道长题其实就能正常落盘，不会掉入 fallback。

第二层是强约束输出，也就是 Structured Outputs。它的关键不在于“提示词写得更狠”，而在于生成阶段就把不符合 schema 的输出路径掐掉。说白了，模型不再被允许随手加 markdown、前言后语或者别的废话，而是只能沿着合法结构往下生成。到了这一步，格式漂移就不是“少一点”，而是从机制上被压住。

第三层是函数调用化。这个更适合像我们这种要把结果直接接进业务流程的场景。与其让模型输出一段自由文本，再祈祷你能从里面抠出 `questions`，不如把它变成一次明确的调用，比如 `emit_interview_questions`，参数里就定义好 `questions` 数组、每个字段的类型和必填项。模型不再是在“写文章”，而是在“填参数”。系统看到的不是一段看起来像 JSON 的文本，而是一笔结构化调用。这个工程味儿更重，但也确实更稳。

## 回到这个项目，最该改的不是模型，而是接口纪律

这次踩坑给我的最大提醒，不是“模型偶尔不靠谱”，而是“别把可靠性交给模型自觉”。

这个备面模块本来就不是一个聊天玩具，它是要把 retrieval、prompt、question generation 和 session 持久化串起来的。到了这种链路里，模型输出已经不是“参考意见”，而是下游系统的输入。既然是输入，就该像别的系统边界一样认真对待：

1. 能强约束，就别只靠提示词劝说
2. 能函数调用，就别把结构化结果塞进自由文本
3. 能容错修复，就别一失败就立刻 fallback

说白了，模型内容能力当然重要，但一旦进入工程环境，“守规矩”往往比“会发挥”更值钱。

以前做普通后端，大家都知道接口字段、类型、状态码不能乱来。轮到 LLM 这儿，很多人突然开始佛系，觉得“差不多能看懂就行”。其实哪有这种好事。只要结果还要继续被程序消费，AI 输出就一样得讲纪律。

这事想明白以后，很多问题就都顺了：上面那层是智能，下面这层是约束。智能决定上限，约束决定系统到底会不会半夜把自己炸了。

## 推荐阅读

如果想把这件事再往深里看，下面几篇官方材料都值得读：

- [Structured Outputs | OpenAI API](https://developers.openai.com/api/docs/guides/structured-outputs)：讲清楚什么叫强约束输出，什么时候该用 schema 把模型拴住。
- [Function calling | OpenAI API](https://developers.openai.com/api/docs/guides/function-calling)：如果你不想再从自由文本里抠结构化结果，这篇基本就是正路。
- [Introducing Structured Outputs in the API | OpenAI](https://openai.com/index/introducing-structured-outputs-in-the-api/)：官方对这套能力背后思路的解释，比单纯看参数文档更容易建立整体理解。
