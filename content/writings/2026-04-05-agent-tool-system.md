做 Agent 系统，很多人的第一步往往是疯狂开脑洞：“我要给它接个这，再给它接个那”。总觉得只要把工具箱塞得足够硬核，Agent 的能力天花板就会无限拔高。

但真扔到工程环境跑上一阵子你就会发现：造工具从来都不缺想象力，代码一卷全都是；这里头最缺的东西，叫纪律。

当工具列表越拉越长，最先炸掉的往往不是业务逻辑，而是你的底盘——两个工具同时并发写了同一个文件却没加锁，某个临时起意的 API 悄悄把上下文污染了，测试通过但在生产上权限形同虚设，线上一堆查不出源头的诡异副作用。本来是指望给 Agent 多长出几只手干活，结果因为都不守规矩，硬生生演成了左手打右手的互殴局。

说白了，问题根本不在于 Agent 手里的工具数量有几页，而在于这堆零星的武器到底有没有被系统通过契约严丝合缝地约束起来、有没有被调度器拿规则真正“看懂”。

结合我的实战和网上各路大神的经验，我来总结几个点。

## 协议得卡死，不能靠自觉

一个靠谱的工具，至少得说清楚五件事：**叫什么、接收什么、能不能用、做什么、展示什么**。少一个，这工具就带着坑。

这些东西乍一看像是文档规范，但如果只写在 README 里口口相传，早晚会炸。它们必须被死死烧进类型系统和协议里：

- 叫什么：稳定的 `name` / `tool_id`，用于调用和审计。
- 接收什么：结构化的参数 schema（类型、必填、约束、默认值）。
- 能不能用：启用条件、权限边界、资源配额（比如“只有在当前会话绑定了项目 id 才能调”）。
- 做什么：**语义契约**——是否只读、是否幂等、可能改哪些资源。
- 展示什么：返回结构、错误模型、对上游的可观测字段（用于 UI / 日志 / 监控）。

如果不把这些锁在代码上，后果就是：

- 不按协议声明，**编译期直接报错**；
- 协议声明乱写，**运行时当场被调度和风控按住**。

“大家约定好这么写”和“不这么写就跑不起来”之间的距离，大概就是玩具和工程的距离。

有了这个铁基建，后面的事儿才有得聊。权限怎么拆、怎么并发、怎么调度、怎么记日志，全都是叠在这套协议之上的。如果协议本身就是软的，上面盖再高的楼也没用，早晚连地基一块儿塌。

## 工具不是函数列表，是能力图谱

再往深看一步：大家很容易把工具当成一个大大的“函数列表”，但在 Agent 眼里，它应该是一张**能力图谱**。

每一个扔进去的工具，除了管好自己的输入输出，必须在能力层面把标签打清爽：

- **领域标签**：文件系统、代码库、外部 API、支付、运维……
- **资源作用域**：“只读某个 repo”、“只能在临时目录里折腾”、“只准调这个三方库”。
- **风险等级**：只读、覆写、强删、甚至动钱，不同风险对应不同的风控水位。
- **调用成本**：几毫秒的延迟还是几美元的消耗？

只有把这些底牌交给了调度器，系统才有可能做点聪明事：

- 同领域下的小散工具，顺手就组合成了高阶动作；
- 高危操作不仅被盯死，还能强制限流、留痕、随时回放；
- 那些又慢又贵还不怎么出活儿的工具，调度器直接就能做降级。

如果能力不建模，模型看到的就只是一堆光秃秃的英文名和两段自然语言，最后到了 tool selection 的时候只能闭着眼睛“盲猜”。出了事，大家又转头集体吐槽模型“不够聪明”。这锅模型背得真冤。

## 默认不安全，才是真安全

claude code 的工具有一个属性叫 `isConcurrencySafe`，用来声称自己能不能跟别人同时跑。

这里有个极其关键的设计：这个值**必须默认是 false**。

不是“随手没填就当它是安全的”，而是“没填一律当大危工具处理”。想要吃并发调度的红利，就必须得主动举手声明。这在工程里叫 fail-closed：默认卡死，显式放行。

你想想，如果默认是 true 会怎样？那些从没考虑过并发、随手瞎写的工具，全都会被系统放行去裸奔。过阵子开始经常静默地读到脏数据，又或者两个写操作在互踩并发坑。这些 bug 甚至都不会报错，只是默默把结果搞烂。查都没法查。

但凡涉及安全，系统设计就绝不能建立在“大家都很自觉、都会记得配”这种天真的念头上。人都是会偷懒的，文档是没人看的，代码是会一路复制粘贴的~~，坦克是没有后视镜的~~。默认保守、显式放行，才是唯一能撑住底线的结构。

这种思路一样应该铺到别的地方：

- **权限管理**：默认拦在门外，授权必须精确到资源级；
- **破坏性操作**：默认只能读，想删改就必须显式打上标记如 `destructive` ；
- **外网访问**：默认不通公网，需要什么域名就老老实实配白名单；
- **成本控制**：默认捏死并发度，想提速就提配额申请。

工程系统的责任，就是让那些“没想清楚的半成品”在编译期或测试期就干脆利落地炸响，而不是变成一枚枚定时炸弹悄悄埋在生产环境里。

## 并发分区不是“一起上”，而是“切批次”

很多人在做调度的时候，最容易把“并发”简单粗暴地理解成“所有查出来的工具一波流跑完”。但其实这事儿的关键是：**到底哪些能凑一块儿跑**。

比较落地的做法是按安全属性做“切批”。一连串调用串过来，没副作用的攒成一把并发带走，有副作用的单独摘出来阻塞式执行。碰到危险操作，绝对不再往下偷跑半步。

举个例子：`[Read(a), Read(b), Bash(ls), Read(c)]`。这套连招下来，切完应该是三批：

- 第一批：前俩 `Read` 并发带走；
- 第二批：单独阻塞跑 `Bash`；
- 第三批：等 `Bash` 落听了，再跑最后一个 `Read`。

凭什么第三个 `Read` 不能跟 `Bash` 一起？因为这段 `Bash` 说不定就刚好把 `c` 给改了，你得等文件系统稳了再去读，不然读出来的就是个薛定谔的中间态。

逻辑不难，难在开发工具的人自己到底知不知道这个工具有没有副作用。一个干干净净的纯读操作，你给它报了危险，顶多也就是系统跑得慢了点；你要是把一个带副作用的操作瞎标成安全，那基本就是往自家祖坟上埋雷。

这就是为什么前面说 `isConcurrencySafe` 这类字段绝不能当成文档备注。它就是**直接把控调度命运的物理开关**。在填这个值的时候，你最好确认自己是真的想清楚了。

如果手痒还想卷得更进一步，连“读写哪些具体资源”、“是不是幂等”也全盘托出，调度器简直就能玩出花来：不只是切批，连细粒度的**冲突检测甚至乐观并发**都能做。只要是不重叠的事务，想怎么狂飙怎么狂飙。

## 榨干等待时间：流式调度与顺序纪律

常规的批量调度很笨：非得干巴巴地等模型把所有的 tool call 都吐全了，才慢吞吞地开始按批跑。这里的空窗期，机器就在那儿发呆。

流式调度就是把这段时间重新压榨干：吐出半个合法结构了？好，立刻塞进队列里去跑。等模型吭哧吭哧排版第三个工具的时候，第一拨说不定都已经落盘了。这叫在飞行中换油，整体的延迟肉眼可见地就下来了。

但这招有个铁律必须得守住：**并行跑可以，结果给出来的顺序绝不能乱**。这就好比食堂打饭，后厨可以是并发炒菜的，但给到大堂的叫号顺序必须严丝合缝。工具 B 即便手脚快跑完了，也得老老实实呆在缓冲区，等到 A 稳妥落地了才能一起出去。

这个口子绝对不能松。因为上游模块大概率是按顺序消化东西的。你一旦往上推无序的脏流，上面就得被逼着自己搓一套极其别扭的时序恢复锁；如果不写，行为就会变得非常诡异。把保序这个苦活脏活压死在调度器里，上面才能写得清爽。

这背后的实现其实不复杂，弄个状态机就行：

- 工具刚跑完不叫“完事儿了”，叫处于“已完成待输出”缓冲区；
- 调度器就死死盯着那个“最早没发出的指针”，前面的坑不填满，后面的再急也得憋着。

顺带提个工程细节：虽然最终答案死卡顺序，但**过程里的进度消息（Progress）是随时随地能往外透的**。前端转圈圈的动画需要的只是“系统没挂还在动”的暗示，不需要较劲这转圈分不分先后。最终结果才是斩钉截铁的事实。

稍微跳出来看，这其实就是强行把“模型推理”和“工具执行”揉成了一条指令流水线：结果一边淌回上下文给模型喂新的提示词，一边往前台抛着状态光效。系统的整体延迟，就从原本的串行相加，完美降级成了整条管子里最慢的那部分而已。

## 工具越专一，系统就越稳当

很多人很喜欢造那种“大包大揽”的神仙工具，觉得省事儿。但这玩意儿最致命的问题就是**语义太泥石流了**。

你搞个大一统的 `BashTool`，看着是万能的，但你没法跟调度器解释“我这次真的只是 `cat` 了一下文件，你放行吧”。在调度器眼里，这通操作的内部是完全看不见的黑洞，为了保命，它只能默认你是在格式化硬盘，然后把所有并发都给你掐个粉碎。

最好的办法，反而是把原本的工具强行捏成一个个“窄口专门化”的零件：

- `FileReadTool` 就是个本分的老黄牛，绝对只读；
- `GlobTool` 天生并发安全，爱怎么跑怎么跑；
- `FileEditTool` 永远是有副作用的；
- `GitDiffTool` 只管看，不乱碰；
- `GitCommitTool` 只要动了就肯定改历史，调度器直接清场。

调度器压根不需要花心思猜你里面写了什么代码，看一眼名片就行。

说白了，这也是个**系统可预测性**的博弈。系统只要开始依赖运行时去动态猜你的意图，它就变得极难推理、极难测试。出了 bug 连复现都无从下手。

专门化虽然意味着你要写一堆碎零件，维护起来看着烦，但它的工程红利太稳了：它消灭了所有不可控的“惊喜”。在工程的系统里，“没惊喜”恰恰就是最高的褒奖。

靠一个 `HttpTool` 扫平天下、一个 `SQLTool` 直连所有库、一个 `BashTool` 乱拳打死老师傅，这种搞法一时爽快，最后留下的必定是一个随时会炸的黑盒废墟。

## 把观测和回放武装到牙齿

工具系统讲得再怎么天花乱坠，要是不带可观测探针，就等于瞎子过河。

但凡工具起调了一次，最起码的规矩，就是老老实实落成一条高保真的审计流：

- 谁（哪个 Agent / 会话 / 用户）闲着没事调了这个工具；
- 塞了什么参数进去；
- 当时的上下文、环境变量是怎么配的；
- 最后吐出来的骨头是什么（耗时、真实开销、错误码）；
- 调度器当时脑补的决策是什么（凭什么把它跟前一个并行了，又凭什么拦了另一个）。

这堆日志可不止是拿来排障修 bug 的，它是日后整个系统进化的**血槽和语料库**：

- **离线智能评估**：看看哪个工具天天被拉出来但屁用没有，直接下线斩首或者重写；
- **安全红线盯防**：高危口子要是被无脑猛冲，就得立马补上限流和人机校验；
- **自动宏升阶**：从那些“反复出现的连招技能”里，顺手抽象出一个组合高级工具，直接给模型砍短推导链。

没有探路的探针，你的系统永远是个不知道深浅的黑盒。所谓的设计，全成了 PPT 上的自嗨。

## 工具得治：发现、版本、下线

干上几个月的工具平台，很容易漏掉一个常识槽点：**工具并不是死物，是有生命周期的**。

- **能被发现**：给 Agent 一顿狂堆 System Prompt 来认工具其实挺蠢的。更体面的路子，是做一套带参检和向量检索的基础工具目录。让“定工具”本身也变成一个带约束的工程节点。
- **能扛改版**：工具参数要升、口径要变，调用方咋活？稳稳的做法是：切灰度，挑一部分请求先趟雷；支持拿旧版本的回放录像去测新版有没有掉坑；出了事能立刻秒级切回去。
- **能上就能下**：系统不该是个收破烂的回收站。长期不怎么跑、跑一下动静剧烈还产出一般的冗余工具，早点按头退役，强行留着只会让熵越弄越大。

说直白点，稍微上点规模的工具架，底子里完全就是个“微服务基建”，这得要看注册中心怎么搭、路由怎么甩、限流熔断得立多高。容不得你就是随手丢一把脚本然后双手合十等模型“自己看着办”。

## 题外小结：“手多”不如“手稳”

能不能给 Agent 装几十只手，那是量的问题；它这只干起活来手抖不抖系统到底懂不懂，这是定性底盘的事。

这世上真正硬气的 Agent 工具生态，绝对就是老实本分想明白了这些基建上的坑：

- 协议先行，类型永远比文档口水战强；
- 见代码默认有雷，你自证安全了系统才松口；
- 并发更是精密分组冲突，而不是“不管三七二十一开个多线程”；
- 等待被流式管道强压成低延迟的体验品控；
- 工具收紧狭窄面，专事专做，不求跨界乱套；
- 日志拉满，事件可追溯可全回放。

堆几百个工具出来其实不难。但只有骨架上撑得住“**可预测、全回放、强治理**”的地基，才能叫真的过活在工程环境。剩下的无非是一些定时炸弹状态的技术债。
